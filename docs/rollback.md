# 回滾規範

## 目的

本文件定義部署異常時的回滾流程，目標是快速恢復服務並保留完整可追溯紀錄。

## 回滾觸發條件

- 部署後 production health check 失敗
- 主要使用流程中斷
- 錯誤率超過既定門檻
- 發現安全風險或資料正確性風險
- 部署到錯誤環境或錯誤版本

## 角色責任

- 事件負責人：
  - 判斷是否啟動回滾
- Release Owner：
  - 執行或核准回滾
- 開發者：
  - 協助影響分析與驗證

## 回滾前提

- 已確認最近可用的穩定版本
- 已確認回滾指令或回滾流程
- 已確認受影響環境
- 已建立通知與溝通管道

## 回滾流程

1. 立即停止後續 rollout。
2. 確認最近可用版本或 commit SHA。
3. 依專案核准流程執行回滾。
4. 驗證服務健康狀態與主要業務流程。
5. 確認 log 與告警已回到可接受範圍。
6. 記錄根因與後續改善項目。

## 驗證清單

- 入口頁可正常開啟
- 登入驗證流程正常
- 關鍵 API 回應符合預期
- 背景任務（如有）狀態正常
- 無新增重大告警未處理

## 回滾紀錄要求

回滾紀錄至少包含：

- 事件開始時間
- 問題部署版本
- 回滾目標版本
- 核准者
- 驗證結果
- 後續負責人
